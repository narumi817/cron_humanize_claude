# 今後の改善案

## セキュリティ

### Brakeman を導入する

理由:
- Rails専用のセキュリティスキャナーであるため
- 公開前のセキュリティチェックとして有用
- 保守コストが低い

後回しにしている理由:
- 現在はMVPを素早く公開することを優先しているため
- 現時点では攻撃対象となる範囲が小さいため
- MVPフェーズでは検証処理を軽量に保ちたいため

実行コマンド例:

docker-compose run --rm web bundle exec brakeman

## UX改善

### スマホ表示での書式ガイドラベルのズレ

背景:
- 「分・時・日・月・曜日」ラベルを flex で均等分割しているが、入力値（monospace フォント）の実際の位置とずれて見える
- PC より画面幅が狭いスマホで特に顕著

改善案:
- 入力欄の各フィールドに対応した位置にラベルを表示する
- テーブルレイアウトや固定幅での整列を検討する

考慮事項:
- 入力値の文字幅はユーザーの入力に依存するため、完全な対応は難しい
- JS なしでの完璧な整列は困難なため、実装コストと見た目のトレードオフを要検討

### エラー時にどのフィールドが無効かを表示する

背景:
- 現在のエラーメッセージは「cron式の形式が正しくありません」と書式例のみ
- 「0 0 0 0 0」のような入力でも理由がわからず修正しにくい

改善案:
- 入力値を分析して「日（3番目の値）は 1〜31 の範囲で入力してください」のように案内する
- day=0 や month=0 など範囲外の値を検出してフィールド名付きで表示する

考慮事項:
- Fugit がバリデーションを担っているため、独自で範囲チェックのロジックを追加する必要がある
- フィールドごとの検証は実装コストがある程度かかる

## MCP導入候補リスト
### GitHub MCP（Issue・PR 管理）

概要:
- GitHub の Issue・PR・コメントなどを Claude から直接操作できる

導入理由:
- BACKLOG.md で管理しているタスクを GitHub Issues と連携させることで、実装 → Issue クローズ → コミットの流れを一貫して扱えるようになる

メリット:
- Issue の作成・クローズ、PR のオープン・レビューコメント確認などを Claude から操作できる
- BACKLOG の整理・移動が楽になる

デメリット:
- 現状の BACKLOG.md 運用で十分回っているなら恩恵が薄い
- ソロ開発では過剰になる可能性がある

---

### Fetch MCP（外部URL取得）

概要:
- 指定した URL のコンテンツを取得できる。本番環境のレスポンスや HTML を Claude が直接確認できる

導入理由:
- デプロイ後の OGP タグ・meta・hreflang などの出力を Claude が実際に検証できるようになる

メリット:
- 「本番で正しく動いているか」を Claude が自律的に確認できる
- 軽量でセットアップが簡単

デメリット:
- Claude Code にはすでに WebFetch ツールがあるため追加価値は限定的
- Render の無料プランでスリープ中の場合、レスポンスが遅い

---

### Sentry MCP（エラー監視）

概要:
- Sentry に蓄積されたエラーログを Claude が参照できる。発生件数・スタックトレース・影響ユーザー数などを確認できる

導入理由:
- 本番で実際に起きているエラーをコードと紐づけながら調査・修正できるようになる

メリット:
- 「どのcron式でエラーが多発しているか」などを Claude が直接把握して修正につなげられる
- エラーの再現手順をコンテキストとして持てる

デメリット:
- Sentry 自体の導入がまだのため、MCP より先にサービス設定が必要
- アクセスが少ない現状では収集データも少なく恩恵が薄い

---

### Sequential Thinking MCP（思考の構造化）

概要:
- 複雑な問題を段階的に分解・整理しながら推論を進めるための MCP

導入理由:
- バグの原因究明や設計の検討など、複数の選択肢を比較しながら進める場面で推論の質が上がる

メリット:
- 複雑な問題に対して Claude が手順を明示しながら丁寧に考えるようになる
- 見落としが減る

デメリット:
- このプロジェクトはシンプルな Rails アプリなので、現状の Claude の能力で十分対応できており導入コストに見合う場面が少ない
- 応答が長くなりやすい

### 日本語 → cron 変換　✅

例:
- 毎日9時
- 平日の18時
- 毎週月曜

実装方針:
- まずは UI ビルダー（ラジオボタン・セレクトボックス・テキスト入力）で実装する
- 自然言語入力は言語の揺れが多く AI なしでは精度が出しにくいため後回し
- UI ビルダーが完成した後、アクセスが増えたタイミングで自然言語対応を検討する

UI ビルダーの利点:
- 分・時・日・月・曜日の構造を操作しながら覚えられるので cron 初心者にも教育的
- AI 不要でコストゼロ、実装がシンプル
- 将来的に自然言語を乗せる基盤にもなる

面白い理由:
- cron humanize の補完機能になる
- 初心者にも使いやすい
- SEOとの相性が良い

completed at: 2026/06/11

### タイムゾーン対応（日本語 → cron 変換の拡張）✅

背景:
- サーバーが UTC で開発が JST の場合、時間のずれを手動で計算する必要がある
- 例: 「毎日9時（JST）に実行したい」→ UTC サーバーでは `0 0 * * *` になる

アイデア:
- 日本語 → cron 変換時にタイムゾーンを選択できる UI を追加
- 選択したタイムゾーンに合わせて cron 式に変換する

後回しにしている理由:
- 「日本語 → cron 変換」の基本機能が未実装のため、まずそちらを優先する
- タイムゾーン対応は UI・ロジックともに複雑度が上がる

completed at: 2026/06/15
