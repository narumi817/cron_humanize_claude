# 今後の改善案

## セキュリティ

### Brakeman を導入する

理由:
- Rails専用のセキュリティスキャナーであるため
- 公開前のセキュリティチェックとして有用
- 保守コストが低い

後回しにしている理由:
- 現在はMVPを素早く公開することを優先しているため
- 現時点では攻撃対象となる範囲が小さいため
- MVPフェーズでは検証処理を軽量に保ちたいため

実行コマンド例:

docker-compose run --rm web bundle exec brakeman

## UX改善

### エラー時にどのフィールドが無効かを表示する

背景:
- 現在のエラーメッセージは「cron式の形式が正しくありません」と書式例のみ
- 「0 0 0 0 0」のような入力でも理由がわからず修正しにくい

改善案:
- 入力値を分析して「日（3番目の値）は 1〜31 の範囲で入力してください」のように案内する
- day=0 や month=0 など範囲外の値を検出してフィールド名付きで表示する

考慮事項:
- Fugit がバリデーションを担っているため、独自で範囲チェックのロジックを追加する必要がある
- フィールドごとの検証は実装コストがある程度かかる
