# プロジェクトの目的

cron式を人間向けに変換するMVPを作る

# 開発方針

- MVPは小さく保つ
- 最適化より先に完成させる
- 過剰設計を避ける
- 不明点がある場合は推測で実装せず確認する
- 大きな変更前には実装方針を共有する

# ドキュメント

- 設計ドキュメントは `/docs` 配下に作成する
- Markdown形式で作成する
- ファイル名は機能名にする
- ドキュメントは小さく実用的に保つ

# Rails

- controllerは薄く保つ
- ビジネスロジックはservice objectに切り出す
- Railsの慣習を優先する

# フロントエンド

- ERBのみ使用する
- JavaScriptは最小限にする
- UIはシンプルにする

# 品質

- 実装した機能には必ずテストを追加または更新する
- タスク完了前にテストを実行する
- タスク完了前にRubocopを実行する
- Rails Ruby Style Guideに従う
- まずはRails標準機能を優先して使う
- 不要なgemは追加しない
- 可読性と保守性を優先する

# 作業フロー

実装を始める前に:
1. 実装計画を説明する
2. タスクをコミット単位の小さなステップに分割する
3. リスクがある箇所や過剰設計になりそうな部分を明示する
4. 実装開始前に確認を取る

# 実装スタイル

- 段階的に実装を進める
- 30分ごとに区切りよく止められる構成を優先する
- 1コミットにつき1つの論理的変更にする
- 一度にすべてを実装しない

# 検証ルール

コードを変更した後は、毎回以下を実行してください:

1. rubocop を実行する
2. rails test を実行する
3. エラーや失敗がある場合は、次に進む前に修正する

Docker環境では以下のコマンドを使用してください:

- docker-compose run --rm web bundle exec rubocop
- docker-compose run --rm web bundle exec rails test

以下の条件を満たしていない限り、実装完了とは判断しないでください:

- rubocop が成功していること
- test が成功していること
